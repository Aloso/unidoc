What is Unidoc?

Unidoc is a new Markup language, inspired by AsciiDoctor, but syntactically more similar to Markdown. This is useful because Markdown is easy to read and to learn. However, Unidoc tries to make Markdown's syntax even simpler, while also introducing new, powerful features such as tables and macros.

Tutorial

Read on to get a deeper understanding of Unidoc and of how it differs from Markdown. This tutorial contains code snippets. If you have JavaScript enabled, you can modify them and see the result on the rightbelow.

Headings

Headings are prefixed with 1 to 6 number signs (#), followed by a space: 

# Heading 1
## Heading 2
### Heading 3
#### Heading 4
##### Heading 5
###### Heading 6

Alternatively, headings can be underlined with equals signs or dashes: 

Heading 1
=========

Heading 2
---------

Differences to Markdown

In Markdown, headings can have number signs on both ends (i.e. ## Heading ##). This is not allowed in Unidoc.

In Unidoc, underlined headings require at least 2 equals signs or dashes.

Paragraphs

This is a paragraph of text.
Single line breaks are ignored.

Paragraphs are separated with a blank line.

Blockquotes

Blockquotes are indented with a > marker: 

> This is a blockquote.
>
> It contains multiple paragraphs.

Blockquotes can contain anything: 

> ### A heading
> - A list
> > A nested blockquote

Differences to Markdown

The > marker can be omitted on blank lines: 

> A paragraph

> A paragraph in the same blockquote

To separate blockquotes, you can add a limiter ($) between them: 

> A paragraph
$
> A paragraph in another blockquote

In Unidoc, quote markers (>) must be correctly indented: 

 > A blockquote
  > Incorrectly indented line

Lists

Lists can be ordered:

1. An ordered list
2. With two items

Or unordered:

- An unordered list
- With two items

Ordered lists can use a dot (1.) or a closing parenthesis (1)). Unordered lists can use dashes (-), stars (*) or plus characters (+).

List items can contain anything:

+ An unordered list
+ With a nested list:
  * This is a nested list
  * > And here's a blockquote
    >
    > Make sure the indentation is correct!

Note that a list marker must always be followed by a space, and subsequent lines must be properly indented.

Differences to Markdown

Unidoc doesn't support laziness; as soon as a line isn't indented far enough to the right, the list item ends: 

  - The list item
    continues on this line
  But this is not part of the list item!

The width of a list marker does not depend on the number of spaces after the marker: 

-   A list item
  - A nested list item

An ordered list can interrupt a paragraph, even when it doesn't start with 1: 

A paragraph interrupted by
4. A list item

Inline formatting

By default, the following inline formatting modes are supported: 

*Italic*, _also italic_,
**bold**, __also bold__,
^superscript^, #subscript#,
~strikethrough~ and `inline code`.

Inline code is a special case, because its content isn't formatted by default: 

`Absolutely **no** formatting!`

When the inline code contains backticks, use a different number of backticks as formatting delimiters: 

``This: ` is okay, as is this: ``` ``

Formatting delimiters can appear in the middle of a word, except for underscores: 

- n^x^
- **B**old first letter
- __T__his doesn't work

Links and images

Links and images consist of a description in square brackets, followed by a URL in parentheses: 

[This is a link](https://www.google.com).

![An image](https://raw.githubusercontent.com/Aloso/unidoc/main/doc/unidoc.svg)

The parentheses can also contain a title in double quotes, which appears when the cursor hovers over the link/image: 

[This is a link](https://www.google.com "This is a title").

Differences to Markdown

Link reference definitions are currently unsupported.

Thematic breaks

Thematic breaks are rendered as a horizontal line:

------------------

Thematic breaks can consist of dashes (-) or stars (*), and must be at least 3 characters long.

Differences to Markdown

Thematic breaks can't have internal spaces: 

This is a weird list, not a thematic break:

- - -

Code blocks

Code blocks start and end with a code fence. A code fence is a line with at least three backticks (`) or tildes (~): 

```
Code block
```

The closing code fence must have at least as many backticks or tildes as the opening code fence. Backticks and tildes can't be mixed.

As in inline code, no formatting is performed in code blocks: 

``````
**Not bold**
[Not a link](#)

```
```
``````

Code blocks are ideal for programming code. The programming language can be written after the opening code fence, to support syntax highlighting: 

```rust
fn foo() -> &'static str {
    "Hello, world!"
}
```

Comments

// Comments start with two slashes.
// They are NOT displayed, so they're
// perfect for editorial notes.

Differences to Markdown

Unidoc deliberately does not support indented code blocks, only fenced code blocks.

The closing code fence must have the same indentation as the opening code fence.

Tables

Tables are not part of the CommonMark specification, but there's a well-supported syntax for tables in many Markdown parsers. Unidoc uses a different, more powerful syntax.

Each table row starts with two vertical bars, table cells are separated with a single vertical bar: 

|| This is   | a table
|| with four | table cells

A table cell can be prefixed with

  • A number sign (#) to make it a header cell
  • A <, ^ or >, to align it left, centered or right, respectively
  • An additional <, ^ or >, to vertically align it at the top, middle, or bottom, respectively:
||#< These   |#  are header cells
||^  Center  |>  Right       | <br><br>
||<> Bottom  |>< Right-top   | <br><br>

A table cell can also be prefixed with {r} or {r;c}, where r is the number of rows and c is the number of columns the cell should span: 

|| Lorem      | ipsum | dolor | sit
|| amet,      |^{2;3} consetetur
|| sadipscing

HTML

HTML can be used anywhere: 

<kbd>Ctrl+K</kbd>
<span class="hello">world</span>

<style>
  .hello {
    color: orange;
  }
</style>

Macros

The implementation of macros is incomplete and doesn't work yet.

There are macros for a variety of uses. Macros always start with a @, optionally followed by an all-uppercase word, optionally followed by arguments in parentheses: 

@THIS(is a macro)
@AS_IS_THIS
@(and this)

A macro must always be attached to a block or element: 

@(a)
# Heading

@(b)
Paragraph

@(c)
- List
- List
- List

@(d){Text enclosed in curly braces}

@(e)`Inline code`

@(f)<kbd>An HTML element</kbd>

@(g)[A link](https://www.google.com)

You can find a list of available macros further down.

Escapes

Special characters can be escaped with the backslash (\): 

\### This is not a heading

\> This is not a quote
and \**this is not bold**.

\|| This is not a table

\@FOO{This is not a macro}

To escape an image, if you add a backslash before the !, it will be treated as a link: 

\![Alternative text](url)

Instead add the backslash before the brackets:

!\[Alternative text](url)

Numbers can't be escaped. Therefore, to escape an ordered list, put the backslashes before the dots or parentheses: 

1\. Not a list
2\. Not a list

Limiter

The limiter, $ isn't rendered, ever. It's just and indication that parsing should be interrupted. Example: 

~ This ~ isn't formatted as strikethrough because the word is surrounded by whitespace.

But~$ this $~ is!

The limiter can also be used to split a blockquote or list into several parts: 

> A blockquote
$
> Another blockquote

1. A list
$
15. Another list

The limiter can make a blank line non-blank to connect paragraphs: 

This paragraph
$
continues here.

Since HTML elements are considered inline only if there's text in the same line after the opening tag, a $ can help: 

<div style="font-size: 85%">$
    This is considered inline!
</div>

Substitutions

Some characters are automatically substituted. For example, quotes, apostrophes, 3 dots, (R), (C) and (TM). More substitutions can be added via plugins.

This is not yet implemented.

List of available macros

@()

The macro with no name is used to add HTML attributes to elements. For example, the following adds a CSS class, id, or title attribute to a code block: 

@(.cls #id)`Some code`

@(title="hello world")`Hover me`

It can also be applied to HTML:

@(title="hello world")<span>Hover me</span>

The above is equivalent to using braces:

@(title="hello world"){Hover me}

@PASS

The implementation of this macro is incomplete.

With @PASS, formatted text is parsed even in inline code and in code blocks: 

``````unidoc
@PASS
```
This is **bold**
```
``````

@PASS`This is **bold**`.

You can specify which kinds of syntax is parsed in the block. For example, @PASS(macros,inline) or @PASS(@,i) only parses macros and inline formatting. Here's the full list:

  • inline, i for inline formatting (bold, italic, strikethrough, etc.)
  • codeblock, c for code blocks
  • heading, h for headings
  • tbreak, b for thematic breaks
  • subst, s for text substitutions such as `->` to →
  • list, l for lists
  • limiter, $ for the limiter
  • macro, @ for macros
  • math, % for math blocks
  • table, | for tables
  • quote, > for blockquotes
  • html, < for HTML elements

@NOPASS

The implementation of this macro is incomplete.

@NOPASS disables all formatting in the block, so it behaves like inline code and code blocks.
